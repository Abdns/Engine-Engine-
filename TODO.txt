VULKAN - ДОРОЖНАЯ КАРТА
=======================

Путь от нынешнего forward-рендера одного объекта до полноценного
Vulkan-рендера. Порядок важен: пункты выстроены по зависимостям и по тому,
что раньше всего заставит работать с настоящим Vulkan, а не с
"OpenGL на минималках".

Отметки: [x] - сделано, [ ] - осталось.


УЖЕ ЕСТЬ (БАЗА)
---------------
[x] Instance, surface, физическое и логическое устройство
[x] Очередь: графическая + презентная
[x] Свопчейн с выбором present-режима (MAILBOX -> FIFO), image views, фреймбуферы
[x] Render pass: цвет + device-local глубина (D32)
[x] Один графический пайплайн, SPIR-V шейдер, push-константа MVP
[x] Вершинные буферы (host-visible, mapped) и реестр мешей (куб, пирамида)
[x] Один командный буфер, один кадр в полёте, семафоры + fence
[x] Пересоздание свопчейна на ресайз / out-of-date
[x] Плоский дата-ориентированный буфер команд рендера


ФАЗА 1 - СТРАХОВКА И КОРРЕКТНОСТЬ КАДРА (первым делом, дёшево, критично)
----------------------------------------------------------------------
[ ] Слои валидации + debug messenger - VK_LAYER_KHRONOS_validation в
    ENGINE_INTERNAL; без этого троса каждая фича отлаживается вслепую.
[ ] Кадры в полёте (2-3) - свой командный буфер, fence и image-available
    семафор на кадр, чтобы CPU перестал ждать GPU; перестраивает весь цикл
    кадра, поэтому делать рано.
[ ] Отложенное удаление ресурсов - уничтожать буферы/изображения только
    когда GPU их уже не читает (очередь удаления по номеру кадра в полёте).
[ ] Надёжное пересоздание свопчейна - свёрнутое окно (extent 0),
    vkDeviceWaitIdle, пересоздание глубины заодно.


ФАЗА 2 - РЕСУРСЫ И СИСТЕМА ДЕСКРИПТОРОВ (открывает текстуры и освещение)
----------------------------------------------------------------------
[ ] Staging + device-local буферы - заливка вершин через временный
    host-visible буфер и vkCmdCopyBuffer в DEVICE_LOCAL вместо мапа.
[ ] Индексные буферы - vkCmdBindIndexBuffer + vkCmdDrawIndexed; сейчас куб
    рисуется 36 невершинами вместо 8 + индексы.
[x] Дескрипторы - descriptor set layout, pool, sets; первый заход -
    uniform-буфер с view/proj на кадр.
[x] Текстуры - изображение (optimal tiling, device-local), загрузка через
    staging, переходы лейаутов барьерами, image view + сэмплер, привязка
    дескриптором. Затащит staging + дескрипторы + барьеры разом.
[ ] Суб-аллокация памяти - не vkAllocateMemory на каждый ресурс (лимит
    ~4096), нарезать из больших блоков или VMA.


ФАЗА 3 - БАРЬЕРЫ И НЕСКОЛЬКО ПРОХОДОВ (навык "явной синхронизации")
------------------------------------------------------------------
[x] Явные pipeline-барьеры - осознанное владение масками стадий и доступа
    и переходами лейаутов.
[ ] Рендер в текстуру - offscreen-фреймбуфер: нарисовать в изображение и
    прочитать его в шейдере.
[ ] Второй проход - карта теней (depth-only) или пост-обработка; заставляет
    читать то, что только что записал.
[ ] Генерация мипмапов - vkCmdBlitImage с барьерами между уровнями.
[ ] MSAA - мультисемпл-цвет/глубина + resolve-вложение.


ФАЗА 4 - МАСШТАБ РЕНДЕРА (пайплайны, батчинг, пропускная способность)
--------------------------------------------------------------------
[ ] Несколько пайплайнов + кэш + сортировка по пайплайну - VkPipelineCache
    и ключ сортировки в render_entry_mesh, чтобы минимизировать смены состояния.
[ ] Инстансинг и per-object данные - vkCmdDraw с instanceCount,
    динамические UBO или SSBO.
[ ] Многопоточная запись команд - вторичные командные буферы, свой command
    pool на поток. То, чего OpenGL не умеет в принципе.
[ ] Компьют-шейдеры - compute-пайплайн, storage-буферы/изображения,
    барьеры compute<->graphics.


ФАЗА 5 - СОВРЕМЕННОЕ И ПО ЖЕЛАНИЮ (упрощает, не обязательно)
-----------------------------------------------------------
[ ] Dynamic rendering (VK_KHR_dynamic_rendering, ядро 1.3) - выкинуть
    объекты render pass'а.
[ ] Extended dynamic state / shader objects - вернуть гибкость состояния,
    сбить взрыв числа пайплайнов.
[ ] Timeline-семафоры - современная синхронизация вместо бинарных
    семафоров + fence.
[ ] Запросы времени и статистики - timestamp-запросы, бюджет памяти,
    лимиты устройства.
[ ] Bindless / descriptor indexing - привязка ресурсов для больших сцен.


ПРИОРИТЕТ
---------
Обязательный минимум "полноценного" рендера - фазы 1-3 (валидация, кадры в
полёте, staging, дескрипторы, текстуры, барьеры). Фаза 4 - когда сцена
разрастётся. Фаза 5 - вкусовщина и продвинутое.

Самый обучающий первый шаг - текстура через дескрипторы: она одна открывает
сразу три подсистемы (дескрипторы, staging, барьеры/лейауты).
